# CardDB Unreal Bundle Source Version Compatibility Report

## Purpose

Added an opt-in, test-only CardDB bundle source-version compatibility matrix for future importer planning.

This validates compatibility policy only. It does not import CardDB data, load production JSON, create zones, migrate schemas, execute effects, mutate game state, generate runtime activation candidates, generate activation legal actions, or change runtime behavior.

## Compatibility Matrix Fields

The test-only matrix lives in `FWBCardDBSchemaValidationOptions`:

- `bEnableSourceVersionCompatibilityValidation`
- `bRequireSourceVersion`
- `TargetSourceVersion`
- `DirectlySupportedSourceVersions`
- `SupportedTransitions`

Default validation behavior is unchanged because compatibility validation is disabled by default.

## Target Source Version Policy

When compatibility validation is enabled, `TargetSourceVersion` must be non-empty and must pass the same safe `source_version` format policy as bundle source versions.

Malformed matrix targets fail with:

```text
source_version_compatibility_matrix_malformed
```

## Direct Support Policy

A bundle passes compatibility if its `source_version` equals `TargetSourceVersion`.

A bundle also passes if its `source_version` appears in `DirectlySupportedSourceVersions`.

Direct support means validation accepts the source family as compatible. No migration, loading, or runtime mapping happens in this pass.

## Transition Support Policy

A bundle passes compatibility if `SupportedTransitions` contains:

- `FromSourceVersion == bundle source_version`
- `ToSourceVersion == TargetSourceVersion`

Transition support is declarative validation only. It does not transform bundle data.

Malformed transition entries fail with:

```text
source_version_compatibility_matrix_malformed
```

## Missing Source Version Policy

If compatibility validation is enabled and `bRequireSourceVersion` is false, an omitted `source_version` skips compatibility checks.

If `bRequireSourceVersion` is true and `source_version` is omitted or empty, validation fails with:

```text
source_version_missing
```

Malformed present values continue to fail through source-version format or hidden-info diagnostics instead of being treated as missing.

## Unsupported Source Version Diagnostics

If the bundle source version is valid but does not equal the target, is not directly supported, and has no transition entry, validation fails with:

```text
source_version_unsupported
```

The diagnostic path is `$.source_version`.

## Unsupported Transition Diagnostics

If the bundle source version appears as a supported transition source but not for the requested target, validation fails with:

```text
source_version_transition_unsupported
```

The diagnostic path is `$.source_version`.

## Export Behavior

Expected export snapshots now cover source-version compatibility success and failure cases.

Compatibility failures export:

- `ok: false`
- empty `dependency_order_card_ids`
- stable compatibility diagnostic code
- sanitized `json_path`

Successful compatibility fixtures export normal dependency order.

Matrix target, direct-support entries, transition entries, diagnostic messages, full source JSON, and source values are not exported.

## Hidden-Token Safety

`source_version` values containing hidden-token patterns fail closed with `hidden_info_policy_violation` before compatibility decisions.

Expected exports and diagnostics do not include the hidden token.

## Confirmations

- this remains test-only

- no production importer was implemented
- no production loader was added
- no production zones were added
- no Core or Runtime source files were changed
- no effects are executed
- no activation candidates or activation actions are generated by production code
- no runtime activation behavior was changed
- no `FWBAction`, `WBActionCodec`, or `WBRules::GenerateLegalActions` behavior was changed
- no UI, response windows, Blueprints, `.uasset`, or `.umap` files were changed
