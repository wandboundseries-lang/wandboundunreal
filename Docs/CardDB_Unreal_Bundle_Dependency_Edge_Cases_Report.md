# CardDB Unreal Bundle Dependency Edge Cases Report

## Purpose

Added test-only CardDB bundle fixtures and expected export snapshots for importer-ready dependency ordering edge cases.

This remains fixture validation only. It is not a production CardDB importer, loader, zone model, runtime activation path, gameplay rule, or EffectRunner path.

## Duplicate Reference Policy

Duplicate valid references to the same dependency do not emit diagnostics.

The dependency card appears once in `DependencyOrderCardIds`, before the dependent card.

## Mixed Reference-Level Policy

References at card, activated-effect, and payload levels all create dependency edges.

The mixed-level fixture verifies card-level, effect-level, and payload-level dependencies all appear before the dependent card.

## Repeated Path Policy

Repeated direct and transitive paths remain stable.

If A depends on B and C, and C depends on B, the expected order is B, C, A.

## Disconnected Component Tie-Break Policy

Disconnected components use the existing zero-dependency input-order tie-break.

The fixture intentionally uses nontrivial input order and snapshots the resulting order:

```text
component_d, component_c, component_b, component_a, component_e
```

Each component still places dependencies before dependents.

## Independent Card Ordering Policy

Cards with no references preserve their input order exactly.

## Invalid And Missing-Reference Skip-Order Policy

Dependency order now fails closed when validation has already emitted diagnostics before dependency ordering.

This keeps `DependencyOrderCardIds` empty for:

- missing references
- duplicate card ids
- invalid bundled cards
- strict unknown-field failures
- malformed bundle-level validation failures

Self-references and cycles also leave dependency order empty after adding their dependency diagnostics.

## Strict And Non-Strict Ordering Policy

Strict unknown-field failures skip dependency ordering.

The equivalent non-strict fixture still computes dependency order when the unknown field is ignored by default validation.

## Hidden-Token Export Safety Policy

Hidden reference values are not echoed in diagnostics or export JSON.

The hidden-reference fixture verifies `SECRET` does not appear in the exported validation result.

## Expected Export Fixture Coverage

Added expected export snapshots under:

```text
Reference/GodotCanon/CardDBSchemaFixtures/ExpectedExports/
```

Coverage includes duplicate references, mixed reference levels, repeated paths, disconnected components, independent cards, repeated effect/payload references to the same dependency, skip-order invalid fixtures, strict/non-strict unknown-field behavior, and hidden-token safety.
